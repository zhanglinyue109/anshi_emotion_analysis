# agents.md

本文件定义本仓库的协作规范，适用于人类开发者与自动化 Agent。

## 1. 仓库目标（核心思想）

围绕“安史之乱与诗歌情感变迁”构建可复现流程：

1. 单句抽取与结构化（`anshi.py`）。
2. 细粒度情感标注（`annotate.py` 及重试脚本）。
3. 年份/作者/地点统计分析（`anshi_count.py`）。
4. 情感句向量化与聚类分析（`emopoem_get.py`、`embedding_get.py`、`kcluster.py`）。
5. 翻译与 VA 回归作为补充实验链路（`translate*.py`、`xlm_*.py`、`va_*.py`）。

## 2. 修改大纲（强制执行顺序）

1. 先改配置，再改业务逻辑
- 统一从环境变量读取 `API_KEY`、`BASE_URL`、`MODEL`。
- 路径参数化（CLI 参数或配置文件），禁止新增绝对路径。

2. 再改数据结构与标签体系
- 涉及列名、标签映射、输出字段的变更，必须同步更新统计脚本。

3. 最后改模型与实验参数
- Embedding、聚类、VA 模型改动必须保留旧参数可回滚入口。

4. 每次改动必须做最小验证
- 至少跑通 1 个小样本（建议 20 条以内）并确认输出文件可读。

## 3. 开发规范（必须遵守）

1. 安全规范
- 严禁提交明文密钥；发现历史密钥必须视为泄露并轮换。

2. 路径规范
- 仅使用相对路径或可配置路径；禁止 `/home/...` 这类机器绑定路径。

3. 数据规范
- 原始数据只读，不覆盖。
- 中间结果与最终结果分目录保存，文件名带版本或日期后缀。

4. 脚本规范
- 新脚本需提供 `if __name__ == "__main__":` 入口。
- 长流程必须支持断点续跑或定期落盘。
- 失败必须写日志（错误行、错误类型、重试次数）。

5. 文档规范
- 只要主流程输入/输出变化，必须同步更新 `README.md`。
- 新增实验脚本时，在 README 的“如何修改（大纲）”登记影响范围。

## 4. 仓库整理意见（执行建议）

1. 以 `anshi_emosion_paper_release/` 为稳定主线，其他历史脚本归档到 `legacy/`。
2. 新建统一结构：
- `src/pipeline`（主流程）
- `src/experiments`（探索实验）
- `data/raw|interim|processed`
- `outputs/figures|tables|logs`
3. 增加：
- `requirements.txt`（或 `pyproject.toml`）
- `.env.example`
- `.gitignore`（屏蔽大模型权重、缓存与大中间文件）

## 5. Agent 执行清单

1. 改代码前先确认输入文件、输出文件、依赖是否齐全。
2. 改动后至少验证：脚本可启动、输出列完整、编码为 UTF-8（必要时 UTF-8-SIG）。
3. 提交说明必须包含：
- 改了什么
- 为什么改
- 如何回滚
- 哪些输出会受影响
